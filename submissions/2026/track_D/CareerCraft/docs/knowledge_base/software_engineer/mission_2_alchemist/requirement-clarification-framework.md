# 需求澄清框架：把模糊需求变成可执行 API

## TL;DR

- **核心规则**：用 5W2H 与模糊词清单，把"我想要 X"翻译成可量化、可实现的需求列表，再开始设计 API。
- **反例**：产品说"帮我做个用户列表接口"，你立刻动手——结果分页方式、排序字段、权限范围全部猜错，返工 3 轮。
- **在本任务里怎么用**：参照 §3 与 Amy 进行 2–3 轮追问，把每轮问答写进 `requirement_clarification_log.md`，再据此输出 `api_design.md`。

## 1. 为什么要澄清

新接到的需求几乎总是不完整的。直接动手会带来三种成本：

- 返工：实现完才发现误解了字段含义、分页规则、权限边界。
- 摩擦：交付后产品/前端不满意，沟通成本飙升。
- 隐性 Bug：边界情况（空、超长、未授权）没人定义，上线后出问题。

## 2. 5W2H 框架

每个新需求至少回答这 7 个问题：

| 维度 | 关键问题 | 在 API 设计里的影响 |
| :--- | :--- | :--- |
| Why | 为什么要做？解决谁的什么问题？ | 决定要不要做、优先级 |
| Who | 谁来调用？谁是数据所有者？ | 鉴权、权限范围、租户隔离 |
| What | 返回什么字段？字段含义？ | Response schema |
| When | 实时还是异步？数据时效？ | 缓存、轮询、Webhook |
| Where | 来源系统？目标系统？ | 数据源、上下游依赖 |
| How | 怎么调用？分页/排序/筛选？ | Query 参数、Pagination |
| How much | 数据量级？QPS？SLA？ | 性能、限流、索引 |

## 3. 与 Amy 的澄清示例（3 轮追问）

**Amy 原始需求**："我要一个用户列表接口，能查所有用户。"

### 第 1 轮：定位 Why / Who

- 你问：这个列表给谁用？是后台管理、客服查询，还是 C 端展示？
- Amy 答：后台运营人员，做活动定向。
- 你问：运营能看到全部用户，还是只看自己负责的渠道？
- Amy 答：只看自己负责的渠道。
- **收获**：需要鉴权 + 渠道过滤。

### 第 2 轮：定位 What / How

- 你问：列表里需要哪些字段？
- Amy 答：用户 ID、昵称、注册时间、最近一次下单时间、累计消费金额。
- 你问：要不要按某个字段排序？有筛选吗？
- Amy 答：默认按"最近一次下单时间"倒序；要能按"注册时间区间"和"累计消费金额阈值"筛选。
- **收获**：响应字段清单 + 排序/筛选参数。

### 第 3 轮：定位 How much / 边界

- 你问：单个渠道大约多少用户？一次拉多少条？
- Amy 答：每个渠道 1–10 万；一屏 50 条够用。
- 你问：如果某个用户没下过单，"最近下单时间"显示什么？
- Amy 答：显示"-"，排序时排到最后。
- **收获**：分页方式（基于游标更稳）、空值处理。

## 4. 模糊词清单（看到必追问）

- "所有 / 全部"：是真的全部，还是按某个范围？
- "实时"：秒级、分钟级，还是当天即可？
- "支持搜索"：精确匹配还是模糊匹配？多字段还是单字段？
- "权限控制"：基于角色、基于资源所有者，还是两者结合？
- "导出"：CSV / Excel？同步下载还是异步生成？
- "尽快 / ASAP"：要个具体日期。

## 5. 澄清日志模板

```markdown
## 需求澄清日志：用户列表接口

### 原始需求
（一句话原文）

### 澄清记录
| 轮次 | 问题 | 答复 | 影响 |
| :--- | :--- | :--- | :--- |
| 1 | 谁是使用者？ | 运营，按渠道隔离 | 加鉴权 + 渠道过滤 |
| 2 | 返回哪些字段？ | … | Response schema |
| 3 | 数据量？分页方式？ | 单渠道 ≤10 万 | 游标分页 |

### 最终确认的需求清单
1. …
2. …

### 未澄清的问题（待跟进）
- …
```

## 常见误区

- 用"假设"代替"追问"。任何假设都要写下来并让 Amy 确认。
- 一次问 10 个问题，对方答不动。一轮聚焦 2–3 个核心问题。
- 只问"做什么"，不问"为什么"。Why 决定了能不能砍掉某些复杂度。
